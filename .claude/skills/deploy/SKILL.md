# Публикация приложения на Zerovibe

> 🚫 **ЯЗЫК С ПОЛЬЗОВАТЕЛЕМ.** Публикация — место, где легче всего сорваться в
> технический жаргон. НЕ пиши пользователю `Kubernetes`/«кластер»/«под»/«контейнер»/
> «образ»/«сборка образа»/«деплой»/«TLS»/«домен»/«реестр». Для него это неизвестные
> слова — он не понимает и пугается. Вся внутренняя кухня — ТВОЯ. Ему говори только про
> результат: «публикую…», «готово, приложение доступно по ссылке: …». (См. skill
> `conventions`, раздел «Общение с пользователем».)

Платформа собирает приложение из твоего кода (Dockerfile проекта), запускает его и
выдаёт адрес на `zerovibe.app` с авто-TLS. Тебе НЕ нужно собирать архив или заливать
код — платформа берёт актуальный код проекта сама. Твоя работа: проверить код, запросить
публикацию одним вызовом и завести первого админа.

## Как ты общаешься с платформой (важно — облачный режим)

Ты работаешь внутри платформы, поэтому НЕ используешь ключ вайбкодера и НЕ вызываешь
`/dev/projects/...`. Публикация идёт через служебные переменные окружения, которые уже
есть в твоём окружении:

- `ZV_AGENT_DEPLOY_URL` — эндпоинт публикации этого проекта.
- `ZV_AGENT_CALLBACK_KEY` — ключ авторизации (заголовок `X-API-Key`).

Если `ZV_AGENT_DEPLOY_URL` пуст — публикация из этого окружения недоступна (локальный
прогон); скажи пользователю, что опубликовать сейчас нельзя.

## Шаг 0 — проверить ДО публикации

1. **Собери и прогони тесты:** `make check` (или `go build ./... && go test ./...`).
   **Красное — НЕ публикуем**, сначала чини (см. skill `testing-rules`). Пользователю —
   продуктово: «проверил, всё работает» / «нашёл проблему, чиню».
2. По желанию подними приложение и прогони основной сценарий сам для самопроверки
   (см. skill `run`) — но пользователю localhost-ссылку НЕ давай, у него нет доступа к
   твоему окружению. Предпросмотр пользователю обеспечивает платформа.

## Шаг 1 — запросить публикацию

Один вызов:

> `POST $ZV_AGENT_DEPLOY_URL` с заголовком `X-API-Key: $ZV_AGENT_CALLBACK_KEY`,
> тело `{"subdomain": "<sub>"}`.

Поддомен `<sub>`: только `a-z`, `0-9`, дефис; 1..63 символа; уникален на платформе.
Спроси у пользователя желаемое имя, если не задано (через `ask_user`). При
переопубликовании — тот же поддомен, что и в прошлый раз.

Ответ содержит: `domain` (полный адрес, напр. `<sub>.zerovibe.app` — бери его ОТСЮДА,
не хардкодь), `status`, и при ПЕРВОЙ публикации — `setup_url` + `setup_token`.

**Публикация асинхронная.** Сразу после запроса `status` обычно `pending` — это норма:
приложение собирается и раскатывается (десятки секунд — пара минут). Открывай
`https://<domain>/healthz` и жди `200` (готово). Статусы:
- `pending` / `building` → ещё собирается, подожди и проверь `/healthz`.
- `running` → готово, открывай `https://<domain>`.
- `failed` → занятый поддомен, красные тесты или ошибка сборки.

## Шаг 2 — первичная настройка (создать первого админа)

При ПЕРВОЙ публикации в приложении ещё нет администратора — заведи его аккаунт **сам,
запросом**. Пользователь не программист и НЕ открывает ссылки, не вводит коды. Платформа
вернула в ответе публикации `setup_url` (вида `https://<domain>/setup`) и `setup_token` —
бери их ИЗ ОТВЕТА, это для тебя, не для показа человеку.

1. **Спроси у пользователя только продуктовое:** какой email и пароль сделать для входа
   владельца (через `ask_user`). Про токен/URL не упоминай.
2. **Отправь запрос сам:** `POST {setup_url}` с телом
   `{"email": "<email>", "password": "<пароль>", "token": "<setup_token из ответа>"}`.
   - `201` → готово: «завёл тебе вход, открывай приложение и заходи».
   - `410` → админ уже есть (настройка была ранее) — просто скажи войти.
   - `403` → код не подошёл (редко) — переопубликуй и повтори с новым `setup_token`.

> НЕ давай человеку ссылку «зайди и заведи сам», НЕ показывай `setup_token`/`setup_url`.
> Запрос /setup — твоя работа, человек только называет желаемые email и пароль.
> `setup_token` — служебный код: НЕ вписывай его в свои сообщения/ответы в чат (он
> используется ровно один раз в запросе /setup и больше нигде не нужен).

## Что знать (факты)

- Данные приложения (SQLite) — на сетевом диске, переживают переопубликование. Первый
  админ, заведённый на Шаге 2, остаётся и после следующих публикаций (setup_token тогда
  вернёт `410` — это нормально, повторно заводить не надо).
- Платформенные переменные (`PLATFORM_API_URL`/`PLATFORM_API_KEY` для писем,
  `APP_BASE_URL`, `SECURE_COOKIE`, `SETUP_TOKEN`) платформа кладёт в приложение сама.
- Миграции применяются при старте приложения автоматически.

# Экран в два захода: сначала вид, потом логика

Пользователь — создатель продукта, не программист. Ему важно СНАЧАЛА увидеть, как
экран выглядит, и сказать «нравится / переделай», и только потом — чтобы он
заработал. Переделывать вёрстку после написания логики дорого. Поэтому любой
новый экран делай в ДВА захода.

## Фаза 1 — вёрстка (внешний вид, без логики)

Собери экран как HTML-шаблон, который красиво выглядит на демо-данных, но пока
ничего не делает. На каждый новый экран — ТРИ вещи:

1. **Шаблон** `internal/transport/web/templates/<screen>.html` (всё плоско в этой
   папке, НЕ заводи подпапки — иначе шаблоны не загрузятся). Вёрстка — DaisyUI
   (skill `daisyui`) + вкус (skill `aesthetics`, обязательно на «сделай красиво»).
   Данные читай через поля (`{{.Title}}`, `{{range .Items}}`) — их наполнит фикстура.
2. **Фикстура** `internal/transport/web/templates/<screen>.fixture.json` рядом —
   демо-данные экрана в JSON (заголовки, 2-3 примера в списках, реалистичные
   значения; даты — строкой формата `2006-01-02T15:04:05Z`). Это данные ТОЛЬКО для
   показа в фазе 1. **Ключи фикстуры = контракт полей**, которые фаза 2 обязана
   заполнить реальными данными.
3. **Запись в манифест** `internal/transport/web/templates/screens.json` — добавь
   строку `{ "name": "<screen>", "title": "Человеческое имя", "file": "<screen>.html" }`.
   (Манифест — реестр экранов проекта, дёшево держать актуальным.) Партиалы
   (кирпичи, layout) в манифест НЕ добавляй.

Переиспользуемые куски (кнопка, карточка, строка списка, поле) — через
`{{define "..."}}` / `{{template "..." .}}` (механизм есть в layout.html). Новый
экран сначала переиспользует существующие define'ы, создаёт только недостающие.

**Покажи экран в живом превью.** Чтобы новый экран был виден в превью (правая
панель показывает СОБРАННОЕ приложение по «/»), подключи его к роуту сразу: добавь
роут и минимальный хендлер, который рендерит шаблон на данных ФИКСТУРЫ (не из БД —
логики ещё нет). Так вайбкодер видит вид экрана в превью, а бизнес-логику навесишь
в фазе 2. CSS пересобирается сам — `make css` звать не нужно. (Про пользователя
говори просто: «сейчас покажу, как выглядит экран» — без слов про вкладки/галерею.)

Затем СПРОСИ, нравится ли вид — простыми словами. Не начинай бизнес-логику, пока
вид не утверждён (просит поправить — правь вёрстку, снова покажи). ask_user для
развилок по виду.

## Фаза 2 — логика (после утверждения вида)

Вид одобрен → навешивай бизнес-логику НА ТЕ ЖЕ файлы, вёрстку не переписывая:

- Слои по образцу `Note` — domain → usecase → repository → transport (см. skill
  `new-feature`, там подробно ЧТО в каждом слое).
- В шаблоне: заглушки заменяй реальными полями (`{{.Field}}`, `{{range}}`),
  добавляй формы/действия и htmx-атрибуты (`hx-post`/`hx-get`/`hx-target` —
  фрагменты без перезагрузки, skill `conventions`), подключай обработчики.
- Каркас разметки, классы, композицию партиалов — НЕ трогай без нужды: пользователь
  это уже утвердил. Меняешь только то, что нужно, чтобы экран заработал.

Ключи `<screen>.fixture.json` — контракт: в фазе 2 бэкенд заполняет ИМЕННО их
реальными данными (типы — как в `pageData`, web.go). Прошёлся по всем ключам
фикстуры — значит ничего не забыл подключить.

## Когда двухфазность НЕ нужна

- Правишь СУЩЕСТВУЮЩИЙ рабочий экран по мелочи (текст, цвет, поле) — просто правь,
  фазы не разводи.
- Чисто бэкендовая задача без нового UI — сразу логика.
- Пользователь явно просит «сделай всё сразу / по-быстрому» — уважай, не навязывай
  фазы.

## Итог

Новый экран → **Фаза 1**: шаблон + фикстура + строка в screens.json + роут с
хендлером на фикстуре, показал в живом превью, утвердили → **Фаза 2**: бизнес-логика
поверх той же вёрстки. Не пиши логику раньше, чем утверждён вид.

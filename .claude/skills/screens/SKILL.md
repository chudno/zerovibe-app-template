# Экран в два захода: сначала вид, потом логика

Пользователь — создатель продукта, не программист. Ему важно СНАЧАЛА увидеть, как
экран выглядит, и сказать «нравится / переделай», и только потом — чтобы он
заработал. Переделывать вёрстку после написания логики дорого. Поэтому любой
новый экран делай в ДВА захода.

## Фаза 1 — вёрстка (внешний вид, без логики)

Собери экран как HTML-шаблон с ФИКТИВНЫМИ данными — так, чтобы он красиво
выглядел, но пока ничего не делал.

- Пиши в `internal/transport/web/templates/<screen>.html` (существующая структура,
  не заводи новых папок). Вёрстка — DaisyUI-классы (skill `daisyui`) + вкус
  (skill `aesthetics`, обязательно на «сделай красиво»).
- Данные — жёстко в шаблоне (заглушки): выдуманные строки, 2-3 примера в списках,
  реалистичные значения. НЕ подключай реальные данные, обработчики форм, htmx-
  запросы — этого пока нет.
- Переиспользуемые куски (кнопка, карточка, строка списка, поле) выноси в партиалы
  через `{{define "..."}}` / `{{template "..." .}}` (механизм уже есть в
  layout.html). Новый экран сначала переиспользует существующие партиалы, создаёт
  только недостающие — библиотека кусков копится сама.
- **Покажи через живое превью** (skill `preview`): экран сразу виден пользователю
  справа. `make css` после правки классов, чтобы стили подтянулись.

Затем СПРОСИ пользователя, нравится ли вид — простыми словами, показав в превью.
Не начинай логику, пока вид не утверждён (если пользователь просит поправить —
правь вёрстку, снова покажи). ask_user для развилок по виду, если они есть.

## Фаза 2 — логика (после утверждения вида)

Вид одобрен → навешивай бизнес-логику НА ТЕ ЖЕ файлы, вёрстку не переписывая:

- Слои по образцу `Note` — domain → usecase → repository → transport (см. skill
  `new-feature`, там подробно ЧТО в каждом слое).
- В шаблоне: заглушки заменяй реальными полями (`{{.Field}}`, `{{range}}`),
  добавляй формы/действия и htmx-атрибуты (`hx-post`/`hx-get`/`hx-target` —
  фрагменты без перезагрузки, skill `conventions`), подключай обработчики.
- Каркас разметки, классы, композицию партиалов — НЕ трогай без нужды: пользователь
  это уже утвердил. Меняешь только то, что нужно, чтобы экран заработал.

Каждая заглушка фазы 1 — это поле, которое фаза 2 обязана заполнить реальными
данными. Прошёлся по всем заглушкам — значит ничего не забыл подключить.

## Когда двухфазность НЕ нужна

- Правишь СУЩЕСТВУЮЩИЙ рабочий экран по мелочи (текст, цвет, поле) — просто правь,
  фазы не разводи.
- Чисто бэкендовая задача без нового UI — сразу логика.
- Пользователь явно просит «сделай всё сразу / по-быстрому» — уважай, не навязывай
  фазы.

## Итог

Новый экран → **Фаза 1**: вёрстка с заглушками, показал в превью, утвердили →
**Фаза 2**: логика поверх той же вёрстки. Не пиши логику раньше, чем утверждён вид.

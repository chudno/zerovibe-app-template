# Самопроверка вживую

Приложение уже **запущено рядом с тобой** — то самое, что человек видит в панели
превью (см. skill `preview`). Оно крутится на `localhost:8080`, читает твой текущий
код и пересобирается само при правках (air). Поэтому проверять «работает ли на
самом деле» ты можешь **прямо на нём** — ничего запускать не нужно.

Это дополняет `make check`: сборка и тесты не ловят «кнопка не туда ведёт», «форма
не сохраняет», «пустой экран». Прогони реальный сценарий сам — по живому
приложению.

> ⚠️ **НЕ поднимай своё приложение** (`go run ./cmd/server`, `make run/dev`, свой
> процесс на :8080). Оно УЖЕ запущено — твой запуск займёт порт 8080, живое
> приложение упадёт, и превью у человека погаснет. Проверяй существующее на
> `localhost:8080`, не плоди второе. (Подробнее — skill `preview`.)

## Когда

После любой ЗАМЕТНОЙ работы — добавил фичу/раздел, сверстал экран, поправил
поведение. Не спрашивай разрешения — просто проверь живое приложение и опиши
человеку словами продукта, что проверил и что работает.

## Как проверять (по живому приложению на localhost:8080)

Приложение уже отвечает. Убедись: `curl -fsS http://localhost:8080/healthz` → `ok`.
(Если не отвечает — вероятно сборка сломана; см. «Если превью застыло» ниже.)

### 1. Первый администратор (если в приложении есть вход)
Создаётся один раз через `/setup`. В песочнице (превью) код настройки **фиксирован:
`preview-setup`** — не ищи его в логе. Заведи демо-админа сам:

```sh
curl -s -X POST http://localhost:8080/setup \
  -d 'email=demo@demo.dev&password=demo12345&token=preview-setup'
```
201 → создан. 410 → админ в этой песочнице уже есть → просто используй те же
`demo@demo.dev` / `demo12345`. Демо-креды для проверки постоянные — при ПУБЛИКАЦИИ
вайбкодер заведёт своего реального админа (skill `deploy`), демо в прод не уезжает.
НЕ придумывай обходов (вставка в БД, «проверю при публикации») — путь выше рабочий.
Входа в приложении нет — пропусти. Подробно про админку — skill `admin`.

### 2. Насеять немного данных
Если без данных смотреть нечего (пустые списки) — создай 2-3 примера через формы
или админку (POST-ами на живое приложение), чтобы было что показать и проверить.

### 3. Прогнать сценарий
Пройди путь, который затронула работа: создать запись → она в списке →
отредактировать → сохранить → изменения на месте. Проверяй запросами к
`localhost:8080` (GET страниц, POST форм) и глазами по HTML-ответу. Для админки —
логинься на `/admin/login`, проверь CRUD. Убедись: нет пустых экранов, битых
ссылок, несохранений.

Нашёл проблему → читай код, чини. air пересоберёт живое приложение сам за пару
секунд — проверь снова тем же curl'ом (перезапускать ничего не надо).

## Хорошая новость про превью

И ты, и человек работаете с ОДНИМ живым приложением. Значит человек может прямо
сейчас через админку или действие в приложении что-то поменять/создать — и это
сразу отразится (общая песочная база). Это удобно для совместной проверки: пусть
человек потыкает, ты подхватишь.

## Если превью «застыло» (показывает старое)

Признак: правки не появляются, `curl` возвращает старое поведение. Почти всегда —
**сломана сборка**: air не смог пересобрать и держит прошлую рабочую версию.
Прогони `make check` (или `go build ./...`), найди и почини ошибку компиляции — air
подхватит и оживит приложение. Не пытайся «перезапустить сервер» руками.

## После проверки

Всё работает → скажи человеку простыми словами, что готово и что именно проверил
(«создание и редактирование записей работает, проверил вживую»). Публикацию он
запустит кнопкой «Опубликовать» (публикуешь ты, skill `deploy`).

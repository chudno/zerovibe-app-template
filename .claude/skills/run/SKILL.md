# Запуск и самопроверка

Ты работаешь в облачном окружении, где приложение можно поднять и открыть в браузере
ТЕБЕ, чтобы проверить работу вживую. Это дополняет `make check`: сборка и тесты не
ловят «кнопка не туда ведёт», «форма не сохраняет», «пустой экран». Прогони реальный
сценарий сам.

> Публикацию приложения (выкладку на поддомен) делает ПОЛЬЗОВАТЕЛЬ сам кнопкой
> «Опубликовать» на платформе. Ты код НЕ деплоишь. Твоя задача до этого — чтобы всё
> было действительно рабочим.

## Когда

После любой ЗАМЕТНОЙ работы — добавил фичу/раздел, сверстал экран, поправил поведение.
Не спрашивай разрешения поднять — просто подними и проверь, затем опиши пользователю
словами продукта, что проверил и что работает.

## Шаги

### 1. Поднять сервер в фоне
Из папки приложения (где `cmd/server`). Запускай в ФОНЕ, чтобы не блокировать сессию.
Своя временная БД-файл, свой `SETUP_TOKEN` (чтобы создать админа):

```sh
SETUP_TOKEN=dev-setup DB_PATH="file:./zerovibe-dev.db" ADDR=":8080" \
  SECURE_COOKIE=false go run ./cmd/server
```
(`make run` тоже поднимает на `:8080`, но через env удобнее задать SETUP_TOKEN и порт.)

Дождись, пока поднимется: `curl -fsS http://localhost:8080/healthz` отдаёт `ok`.
Если порт занят — возьми соседний (`:8081`…).

### 2. Создать первого администратора (если в приложении есть вход)
Дёрни `/setup` сам (создаёт первого админа один раз):

```sh
curl -s -X POST http://localhost:8080/setup \
  -d 'email=<email>&password=<пароль>&token=dev-setup'
```
Email/пароль — любые для проверки (или те, что назвал пользователь). Если входа нет —
пропусти шаг.

### 3. Насеять немного данных
Если без данных смотреть нечего (пустые списки) — создай 2-3 примера через формы/админку,
чтобы было что проверять.

### 4. Прогнать сценарий в браузере САМ
Открой приложение и пройди тот путь, который затронула работа: создать запись → она в
списке → отредактировать → сохранить → изменения на месте. Для админки — войди в
`/admin/login` и проверь CRUD. Убедись, что нет пустых экранов, битых ссылок, несохранений.
Нашёл проблему — читай код, чинь, перезапускай, проверяй снова.

## Грабли

- **Запускай в фоне** — иначе сессия блокируется. Останови процесс, когда закончил
  проверку (`pkill -f 'go run ./cmd/server'` или по порту).
- **SQLite-файл временный** (`./zerovibe-dev.db`) — не коммить (в `.gitignore` по маске;
  если нет — не добавляй в git). При повторном запуске админ уже есть — `/setup`
  второй раз вернёт ошибку, это норма (удали файл БД для чистого старта).
- **Платформенные фичи без ключей** — при запуске без `PLATFORM_API_KEY` письма пишутся
  в лог (не уходят), файлы складываются в каталог `uploads/` вместо S3. Это нормально
  для проверки — флоу работает, см. грейсфул-фолбэки в `platformmail`/`platformfiles`.
- **`SECURE_COOKIE=false`** обязательно (без него cookie сессии не встанет по http).
- **Порт занят** — не убивай чужой процесс молча; возьми другой порт.
- Останови предыдущий сервер перед новым запуском (rate-limit на вход и порт).

## После проверки

Всё работает → погаси сервер и скажи пользователю простыми словами, что готово и что
именно проверил («создание и редактирование записей работает, проверил в браузере»).
Публикацию он запустит сам кнопкой. Если нашёл проблему — почини и проверь снова.

Аргумент пользователя: $ARGUMENTS

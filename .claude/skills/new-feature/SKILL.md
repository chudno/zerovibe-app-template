# Новая фича в Zerovibe

Пользователь — создатель продукта, не программист (см. «Общение с пользователем»
в skill `conventions`). Реши технику сам по этим правилам, **не задавай вопросов
про реализацию**; уточняй только продуктовый смысл и только если он правда неясен.
По итогу опиши результат словами продукта, без жаргона.

Добавляй сущность срезом через ВСЕ слои, по образцу существующей `Note`. Для
сущности `X` (пример: `Task`, `Client`) выполни по порядку:

## 1. domain — `internal/domain/<x>.go`
- Структура `X` (поля, без тегов), `time.Time` для дат.
- Конструктор-валидатор `NewX(...) (X, error)` — все инварианты здесь
  (обязательные поля, лимиты длины) через `ErrValidation`.

## 2. usecase — `internal/usecase/<x>s.go`
- Порт `XRepository` (интерфейс: `Create/List/Get/Update/Delete` — что нужно фиче).
- Сервис `XService` с методами-операциями: валидирует через `domain.NewX`,
  делегирует порту. Без SQL и HTTP.

## 3. repository/sqlite — `internal/repository/sqlite/<x>s.go`
- `XRepo` реализует порт. Записи — через `db.Write`, чтения — через `db.Read`
  (см. skill `conventions`). `INSERT ... RETURNING id, created_at` — чтобы вернуть
  заполненную сущность.
- Схема таблицы — НОВОЙ goose-миграцией, не inline в коде (формат и причина — в
  skill `conventions`). Новая сущность = новый файл миграции.

## 4. transport/web — хендлеры + шаблоны
🚫 ЖЕЛЕЗНОЕ ПРАВИЛО: экран работает БЕЗ перезагрузки страницы, как SPA (см. раздел HTMX
в skill `conventions`). Все действия — через hx-атрибуты, мутации возвращают фрагмент,
НЕ редирект. Проверь готовый экран: ни одно действие не должно перезагружать страницу.
- В `web.go`: маршруты (`GET`/`POST`/`DELETE`) и хендлеры.
  - `GET` раздела — полная страница (прямой заход).
  - мутации — возвращают HTML-ФРАГМЕНТ изменённого элемента (без редиректа/перезагрузки).
  - поиск/фильтр/сортировка/пагинация — `hx-get` с подменой тела списка.
- Шаблоны в `templates/`: `{{define "content"}}` для страницы, `{{define "<x>"}}`
  для фрагмента. HTMX-атрибуты: `hx-get`/`hx-post`/`hx-delete`, `hx-target`, `hx-swap`,
  `hx-trigger`. Никаких `<form method=post>`+редирект.
- **Оформление — компоненты DaisyUI** (`card`, `btn`, `input input-bordered`, `alert`,
  `table`, `badge`, семантические цвета `base-100/base-content/primary/error` …), см.
  skill `daisyui`. Держись стиля существующих шаблонов, не разводи стили вразнобой.
  **После правки классов пересобери CSS: `make css`** (tree-shaking; `make run`/`check`
  делают это сами). Node НЕ нужен. Встроенная админка — со своими стилями, не трогаем.

## 5. composition root — `cmd/server/main.go`
- Собрать: `repo := sqlite.NewXRepo(db)` → `svc := usecase.NewXService(repo)` →
  передать в `web.NewServer(...)` рядом с уже существующими сервисами (notes/auth/
  settings). Если раздел должен быть личным/защищённым — вешай хендлеры под
  `s.requireAuth(...)` (или `s.requireRole(domain.RoleAdmin, ...)` для админских),
  а владельца бери из `currentUser(r).ID` (см. как сделано в заметках).

## 5.5. Админка — зарегистрируй сущность (CRUD из коробки)

В шаблоне есть встроенная админ-панель: новая сущность регистрируется дескриптором и
сразу получает список/форму/CRUD/поиск/фильтры/сортировку/экспорт/файлы. Кратко:
admin-методы репозитория (`<x>s_admin.go`) + дескриптор (`internal/adminres/<x>.go`) +
одна строка `adminres.RegisterX(...)` в `main.go`.

**Подробный порядок, эталон дескриптора, типы полей и чек-лист «полный цикл поля» — в
skill `admin`.** Открой его при регистрации сущности в админке (особенно для полей
bool/file/relation/date — у них есть нюансы, легко получить «декоративное» поле,
которое не сохраняется).

## 6. Тесты (ОБЯЗАТЕЛЬНО — см. skill testing-rules)
- Unit usecase на фейк-репозитории.
- E2E транспорта через `httptest` на временной SQLite (миграции — `db.MigrateUp`).

## Вход / личный кабинет / роли (готово в шаблоне)

Если пользователь просит «вход», «регистрацию», «личный кабинет», «чтобы у каждого
свои данные» — НЕ пиши аутентификацию с нуля, она уже есть. Сделай так:

- **Личные данные**: новую сущность привязывай к владельцу (поле `OwnerID`, фильтр
  по `currentUser(r).ID`) — как заметки. Защити страницы `requireAuth`.
- **Первый администратор**: спроси у пользователя обычными словами email и пароль
  будущего администратора и заведи его. (Способ завести первого админа — по
  текущему механизму платформы `/setup`; для проверки при запуске приложения —
  через настройки/окружение. НЕ заставляй пользователя редактировать файлы.)
- **Открыть регистрацию** (по умолчанию закрыта): только если пользователь прямо
  просит — выставь настройку `allow_signup=true` (через `PUT /admin/settings` под
  admin-сессией). Иначе пользователей заводит администратор.
- **Восстановление пароля** работает «из коробки» через платформенную отправку писем —
  ничего настраивать не нужно (платформа сама прокидывает ключ при деплое).
- **Подтверждение почты** (опционально): если пользователь просит «чтобы подтверждали
  почту» — включи настройку `require_email_verification=true`. Тогда после регистрации
  приходит письмо, до подтверждения вход блокируется. Тоже готово в шаблоне.

## Проверка

Прогони сборку, статанализ и тесты (`make check`), подними приложение и убедись, что
работает. Всё зелёное — фича готова. Опиши результат пользователю словами продукта.

**Проверь вживую сам:** подними приложение (skill `run`) и прогони новый сценарий в
браузере — что реально работает, а не только собирается. Публикацию запускает
пользователь кнопкой «Опубликовать» (она присылает тебе «опубликуй»), а сам деплой и
заведение первого админа делаешь ТЫ через skill `deploy`; твоя задача — чтобы к этому
моменту всё было рабочим.

# zerovibe — эталонный стартер Go + HTMX

Reference-приложение: веб-приложение на **Go + HTMX + SQLite** с чистой
архитектурой. Это образец, по которому генерируются новые фичи. Минимальный, но
полный вертикальный срез: одна сущность (`Note`) проведена через все слои.

Деплой: один Docker-контейнер на инстанс, SQLite в volume. Stateless-логика +
один файл данных — просто поднять, просто бэкапить (копия файла).

## Стэк

- **Go** (stdlib `net/http`, method-pattern роутинг 1.22+).
- **HTMX** — сервер рендерит HTML-фрагменты, клиент их подменяет. Без SPA, без JS-сборки.
- **Чистый CSS** (`static/app.css`) — без UI-библиотек, единая опрятная база стилей.
  Вся статика (`app.css`, htmx) локальная в `internal/transport/web/static/`, вшита в
  бинарь через `embed` и раздаётся с `/static/` — без CDN в рантайме.
- **SQLite** через `modernc.org/sqlite` (чистый Go, без CGO → статический бинарь).
- **html/template** (stdlib), шаблоны встроены через `embed`.
- **Миграции** через `goose` (`internal/platform/db/migrations/*.sql`, встроены `embed`,
  применяются при старте). Схема живёт только здесь.
- **Аутентификация** на `golang.org/x/crypto/bcrypt` (хеширование паролей), сессии в БД
  + httpOnly-cookie.

## Архитектура (чистая, зависимости внутрь)

```
internal/
  domain/              сущности + инварианты + ошибки (только stdlib)
  usecase/             бизнес-логика + порты (интерфейсы репозиториев); зависит от domain
  repository/sqlite/   реализация портов поверх SQLite
  adapter/             внешние адаптеры (напр. platformmail — отправка писем через платформу)
  platform/db/         SQLite + ОЧЕРЕДЬ ЗАПИСИ (единый писатель) + миграции goose
  transport/web/       HTTP/HTML на HTMX; зависит от usecase
cmd/server/            composition root: склейка слоёв + HTTP-сервер
```

Структуры конвертируются на стыках слоёв; `domain` не знает ни о БД, ни о HTTP.

## Ключевые паттерны (по ним генерируются фичи)

- **Очередь записи** (`platform/db`): все INSERT/UPDATE/DELETE идут через
  `db.Write` → единая writer-горутина → нет `SQLITE_BUSY`. Чтения — `db.Read`,
  параллельно. Вызывающий код не думает о блокировках.
- **Срез фичи**: на сущность — `domain` (конструктор-валидатор) → `usecase`
  (порт + сервис) → `repository/sqlite` (реализация порта) → `transport/web`
  (хендлеры + шаблоны).
- **HTMX**: `GET /` отдаёт полную страницу; мутации (`POST`/`DELETE`) возвращают
  ровно тот HTML-фрагмент, который меняется, — htmx подменяет его на клиенте.
- **Ошибки**: доменные ошибки (`ErrValidation`, `ErrNotFound`, `ErrInvalidCredentials`,
  `ErrRateLimited`, …) мапятся в HTTP в одной точке (`transport/web/web.go:fail`).
- **Миграции**: схема — версионированные `.sql` goose в `internal/platform/db/migrations`.
  Новая таблица = новый файл (следующий номер), не правка старых и не inline-DDL.
- **Аутентификация и роли**: пользователи + роли (гость→user→admin), вход/выход,
  сессии (БД+cookie), восстановление пароля письмом, подтверждение почты (опционально),
  рейт-лимиты. Регистрация по умолчанию закрыта — флаг в настройках (`allow_signup`).
  Защита маршрутов: `requireAuth` / `requireRole`. Личные данные привязываются к
  `OwnerID` (как заметки).
- **Настройки приложения**: типизированный реестр (config + secret) в
  `domain/setting.go`, CRUD под admin (`/admin/settings`).
- **Покрытие**: каждый usecase — unit-тест на фейк-репозитории; транспорт — e2e
  через `httptest` на временной SQLite (см. `*_test.go`).

## Запуск

```sh
make run            # локально, БД ./zerovibe.db, http://localhost:8080
make test           # все тесты
make check          # сборка + статанализ + тесты (проверка перед публикацией)
make docker-run     # в контейнере с volume под данные
```

Требуется Go ≥ 1.25. Перед тем как считать работу готовой — прогоняется `make check`
(build+vet+test) и приложение поднимается для проверки в браузере (см. skill `run`).

### Переменные окружения

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `ADDR` | `:8080` | адрес прослушивания |
| `DB_PATH` | `file:zerovibe.db` (в контейнере `file:/data/zerovibe.db`) | путь к БД |
| `APP_BASE_URL` | `http://localhost:8080` | базовый URL (ссылки в письмах). В проде платформа подставляет домен приложения |
| `SECURE_COOKIE` | `true` | флаг Secure у cookie сессии (локально по http задавай `false`) |
| `SESSION_COOKIE` | `zv_session` | имя cookie сессии |
| `PLATFORM_API_URL` / `PLATFORM_API_KEY` | — | адрес API платформы и сервис-ключ для отправки писем. **В проде платформа подставляет их сама при деплое.** Без ключа письмо логируется в консоль (грейсфул-фолбэк для локалки) |
| `SETUP_TOKEN` | — | одноразовый код первичной настройки. Пока админа нет, `POST /setup` с этим кодом создаёт первого администратора (email/пароль задаёт человек). **В проде платформа генерирует и подставляет код при деплое.** Пустой → `/setup` недоступен |

#### Первый администратор

Создаётся через `POST /setup` (поля `email`, `password`, `token`), пока в системе нет
ни одного админа. После создания первого `/setup` закрывается навсегда. Готового сида
из env-кредов нет — пароль выбирает человек, платформа его не видит.

- **В проде:** платформа генерирует `SETUP_TOKEN`, подставляет в окружение контейнера и
  показывает вайбкодеру ссылку на `/setup` после деплоя.
- **Локально:** задай `SETUP_TOKEN` в `.env`, затем открой `/setup` (или `curl -d
  'email=...&password=...&token=...' localhost:8080/setup`).

Настройки приложения (через `PUT /admin/settings` под admin-сессией):
- `allow_signup` — открытая регистрация (по умолчанию выключена);
- `require_email_verification` — требовать подтверждение почты (по умолчанию выключено).
  Если включено: после регистрации приходит письмо со ссылкой; до подтверждения вход
  блокируется («подтвердите почту»), есть повторная отправка с ограничением (1/мин, 5/час).
